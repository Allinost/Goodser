# config 测试从 Mutex 串行化到依赖注入的改造

## 问题

`src/config.rs` 中 `AppConfig::from_env()` 通过 `std::env::var()` 读取系统环境变量构造配置对象。对应的单元测试需要模拟不同的环境变量组合来验证：

- 默认值（env var 不存在时回退）
- 自定义值（env var 设置时生效）
- 必填项缺失时 panic

原测试代码通过 `env::set_var()` 和 `env::remove_var()` 在测试中修改全局环境变量，执行后清理：

```rust
#[test]
fn test_default_host() {
    env::set_var("APP_HOST", "127.0.0.1");
    let cfg = AppConfig::from_env();
    assert_eq!(cfg.host, "127.0.0.1");
    env::remove_var("APP_HOST");
}
```

## 原因分析

### 1. `std::env::set_var/remove_var` 非线程安全

Rust 标准库文档明确标注：

> `std::env::set_var` and `std::env::remove_var` are **not thread-safe**.

多个测试在并行执行时（cargo test 默认多线程），一个测试的 `set_var` 与另一个测试的 `remove_var` 会产生竞态：

```
Thread A: clear APP_HOST          → env: APP_HOST=未设置
Thread B: set APP_HOST=127.0.0.1   → env: APP_HOST=127.0.0.1
Thread A: read APP_HOST           → 读到 127.0.0.1（错误！应为 0.0.0.0）
```

### 2. 容器环境变量污染

项目通过 docker-compose 运行，容器中预设了大量环境变量：

- `APP_HOST=0.0.0.0`
- `APP_PORT=8080`
- `RUSTFS_PUBLIC_URL=https://rfs.hailong.site/rustfs`
- ...

测试依赖于"env var 不存在时使用默认值"的行为，但这些预设值导致测试失败。

### 3. 首次尝试：Mutex 串行化

```rust
fn with_clean_env<F>(f: F) {
    let _lock = ENV_MUTEX.lock().unwrap();  // 全局锁
    clear_all_vars();                       // 清除所有配置 env
    f();                                    // 执行测试
    clear_all_vars();                       // 再次清理
}
```

**问题**：虽然解决了竞态，但 Mutex 强制所有 config 测试串行执行，降低了测试并行度，且治标不治本。

## 解决方案：依赖注入（Dependency Injection）

### 重构 `AppConfig`

将 `from_env()` 拆分为两层：

```rust
// 公开 API - 从真实环境变量读取
pub fn from_env() -> Self {
    Self::from_reader(env::var)
}

// 内部构造函数 - 接受任意 key-value 读取器
fn from_reader<F>(get: F) -> Self
where
    F: Fn(&str) -> Result<String, env::VarError>,
{
    Self {
        host: get("APP_HOST").unwrap_or_else(|_| "0.0.0.0".into()),
        port: get("APP_PORT").ok().and_then(|v| v.parse().ok()).unwrap_or(8080),
        database_url: get("DATABASE_URL").expect("DATABASE_URL must be set"),
        // ...
    }
}
```

### 测试代码使用 mock 闭包

```rust
fn mock_env(pairs: &[(&str, &str)]) -> impl Fn(&str) -> Result<String, env::VarError> + '_ {
    let map: HashMap<String, String> = pairs
        .iter()
        .map(|(k, v)| (k.to_string(), v.to_string()))
        .collect();
    move |key| match map.get(key) {
        Some(v) => Ok(v.clone()),
        None => Err(env::VarError::NotPresent),
    }
}

#[test]
fn test_default_host() {
    let vars = mock_env(&[
        ("DATABASE_URL", "mysql://test@localhost/test"),
        ("RUSTFS_ACCESS_KEY", "test-key"),
        ("RUSTFS_SECRET_KEY", "test-secret"),
        // 不设置 APP_HOST，验证默认值
    ]);
    let cfg = AppConfig::from_reader(vars);
    assert_eq!(cfg.host, "0.0.0.0");
}
```

### 改造效果

| 维度 | Mutex 串行化 | 依赖注入 |
|------|-------------|---------|
| 线程安全 | ❌ Mutex 只是掩盖竞态 | ✅ 无共享可变状态 |
| 并行度 | ❌ 所有 config 测试串行 | ✅ 可完全并行 |
| 测试隔离 | ❌ 仍需 `clear_all_vars` | ✅ 每个测试独立 mock |
| 代码侵入 | ❌ 测试代码修改 env | ✅ 不触碰全局 env |
| 可维护性 | ❌ 泄漏清理逻辑 | ✅ 纯函数式输入输出 |

### 关键原则

测试应**不依赖全局可变状态**。与其测试"当 APP_HOST env var 不存在时 from_env() 返回默认值"，不如测试"当 reader 对 APP_HOST 返回 Err 时，host 字段为 0.0.0.0"。后者函数式、可并行、可推理。
