# 鉴权相关中间件

## 1. auth_middleware — API Key 认证中间件

**文件**: `backend/src/middleware/mod.rs`

### 功能
对请求进行 Bearer Token 认证，验证请求头中的 `Authorization: Bearer <token>` 是否与应用配置的 API Key 匹配。

### 实现方式
- 从请求头中提取 `Authorization` 字段，剥离 `"Bearer "` 前缀。
- 若 token 为空或与 `State<String>` 中存储的 `api_key` 不匹配，返回 `401 Unauthorized`。
- 匹配则放行请求，交由 `next.run(req)` 处理。

### 注册方式
该中间件**当前未注册**到路由中，仅存在定义与单元测试。
注册需使用 `from_fn_with_state` 将 API Key 作为状态传入：

```rust
use axum::middleware::from_fn_with_state;

Router::new()
    .route("/api/example", post(handler))
    .layer(from_fn_with_state(api_key, auth_middleware))
```

若要对全局路由启用，可在 `main.rs` 的 Router 上追加 `.layer()`：

```rust
.layer(from_fn_with_state(cfg.api_key, auth_middleware))
```

### 测试
见 `middleware/mod.rs` 中 `mod tests`：
- `test_auth_middleware_valid_key` — 携带正确 Bearer Token → 200
- `test_auth_middleware_invalid_key` — 携带错误 Token → 401
- `test_auth_middleware_no_header` — 无请求头 → 401

### 依赖配置
- 环境变量 `API_KEY`（默认 `dev-key`），加载于 `backend/src/config.rs`
- 运行配置见 `.env`：`API_KEY=dev-api-key-change-me`

## 2. request_id_middleware — 请求追踪中间件

**文件**: `backend/src/middleware/mod.rs`

### 功能
为每个请求生成或透传 `X-Request-Id`，并创建 tracing span 记录请求的方法、路径和响应状态码。

### 实现方式
- 从请求头中读取 `X-Request-Id`，若不存在则用 `uuid::Uuid::new_v4()` 生成。
- 创建 `tracing::info_span!("request", request_id, method, path)`。
- 请求完成后记录响应状态码。

### 注册方式
已在 `main.rs` 全局注册：

```rust
.layer(axum::middleware::from_fn(request_id_middleware))
```

注：该中间件**不是鉴权中间件**，但与 `auth_middleware` 位于同一文件，且在路由栈中运行在鉴权之前。

## 3. 白名单访问控制（应用层）

### 文件
- `backend/src/handlers/whitelist.rs` — 请求处理器
- `backend/src/models/whitelist.rs` — 数据模型

### 说明
基于 WeChat openid 的权限控制，通过业务 handler 内部实现，**非中间件形式**。
- 角色：`admin` / `member`
- 提供 `/api/checkWhitelist` 端点查询 openid 是否在白名单中

## 4. 错误响应

**文件**: `backend/src/error.rs`

鉴权失败统一返回：

| HTTP 状态码 | code 字段 | message |
|------------|-----------|---------|
| 401        | 40100     | "Unauthorized" |

响应体格式：`{ "code": 40100, "message": "Unauthorized" }`
