## 1. 项目概述

Goodser 是一款面向个人/小型团队的库存管理微信小程序，核心能力包括：

- **多目录库存管理**：用户可创建多个独立的库存目录（仓库），各目录间数据完全隔离。
- **精细编码体系**：采用 `主分区-子分区-序号-数量-状态` 五段式编码，实现分区分类、数量标识与状态追踪。
- **灵活出入库**：支持单独入库、批量入库、搜索入库三种入库方式；支持批量出库与预留出库两种出库方式。
- **预留机制**：出库前可先预留库存，预留期间库存数量被锁定但可随时取消，确认后转为正式出库。
- **标签管理**：支持自定义商品标签，在商品详情/入库页面可直接管理标签，支持按标签筛选商品。
- **NAS 图片存储**：商品图片存储在用户自有的 NAS 设备上，降低云存储成本。

## 3. 存储逻辑设计

### 3.1 库存目录

| 属性 | 说明 |
|------|------|
| 概念 | 用户自行创建的独立仓库空间，类似于"文件夹" |
| 隔离性 | 各库存目录之间的数据（商品、出库单等）完全独立，互不可见 |
| 默认 | 新用户注册后自动创建一个名为"默认仓库"的库存目录 |
| 删除规则 | 仅当目录下无商品时可删除，删除前需二次确认 |
| 数量 | 不限制创建数量 |

### 3.2 分区体系

每个库存目录内包含两级分区：

- **主分区**：用 26 位大写字母（A-Z）标识，最多 26 个主分区
- **子分区**：在主分区下再用 26 位大写字母（A-Z）标识，最多 26×26 = 676 个子分区

> 分区不是预先创建的，而是在入库时由用户选择主分区字母和子分区字母，系统自动按规则生成编码。

### 3.3 库存数量管理

库存数量在编码的第四段体现，但数据库中单独维护精确数值。编码中的数量段仅作快速识别用途，数据库中的 `quantity` 字段为权威数据源。

## 4. 编码规则

### 4.1 编码格式

```
[主分区]-[子分区]-[序号]-[数量]-[状态]
  A-Z    A-Z    0001-9999  0000-9999  A-Z
```

**示例**：`A-B-0001-0015-C`

| 段 | 含义 | 取值范围 | 示例 |
|----|------|----------|------|
| 第1位 | 主分区 | A-Z | A |
| 第2位 | 子分区 | A-Z | B |
| 第3-6位 | 物品序号 | 0001-9999（自增，不足补0） | 0001 |
| 第7-10位 | 物品数量 | 0000-9999（不足补0） | 0015 |
| 第11位 | 状态编码 | A-Z | C |

### 4.2 序号自增与回收

- 新增商品时，在指定的（主分区+子分区）下自动分配下一个可用序号
- **回收机制**：若中间某序号被删除，该序号进入回收池；下次新增时优先使用回收池中最小的序号
- 示例：A-B 分区下已有 0001、0002、0003，删除 0002 后，下次新增分配 0002 而非 0004

### 4.3 数量段更新

- 入库/出库/预留操作导致数量变化时，编码中的数量段同步更新
- 数量段始终反映当前实际库存数量（预留数量不体现在编码中，在界面显示时附加标注）

### 4.4 状态编码

| 编码 | 含义 | 说明 |
|------|------|------|
| A | 正常 | 商品正常在库 |
| B | 预留中 | 商品部分数量被预留 |
| C | 待检 | 商品待检验 |
| D | 损坏 | 商品已损坏 |
| E | 过期 | 商品已过期 |
| F | 停用 | 商品已停用 |
| G-Z | 自定义 | 用户可自定义扩展状态 |

> 状态编码为预设+自定义模式，初始提供 A-F 常用状态，用户可在设置中扩展 G-Z 的含义。

## 5. 业务流程

### 5.1 入库流程

```
┌─────────────────────────────────────────────────┐
│                   入库方式选择                     │
├─────────────┬──────────────┬────────────────────┤
│  单独新增入库  │  批量新增入库  │    搜索导入入库     │
├─────────────┼──────────────┼────────────────────┤
│ 填写商品信息   │ 循环填写多个   │ 搜索已有商品        │
│ 选择分区/状态  │ 商品信息      │ 选择匹配商品        │
│ 上传图片     │ 批量提交      │ 输入新增数量        │
│ 系统生成编码   │              │ 选择分区/状态       │
│ 更新库存数量   │              │ 系统更新编码和库存   │
└─────────────┴──────────────┴────────────────────┘
```

#### 5.1.1 单独商品新增入库

1. 用户填写：商品名称、原价、市场价、预期出售价、库存数量、备注信息、仓储位置
2. 用户上传商品头图
3. 用户选择：主分区（A-Z）、子分区（A-Z）、状态编码（A-Z）
4. 系统自动在该分区下分配下一个可用序号
5. 系统生成完整编码：`[主分区]-[子分区]-[序号]-[数量]-[状态]`
6. 保存商品记录，更新库存

#### 5.1.2 批量新增入库

1. 与单独入库相同的表单，但支持连续添加多个商品
2. 每个商品独立选择分区和状态，独立生成编码
3. 以列表形式展示已添加的商品，支持删除/编辑
4. 确认后批量提交

#### 5.1.3 搜索导入入库

1. 用户输入关键词搜索已有商品（跨当前库存目录）
2. 展示匹配结果列表
3. 选择商品后输入新增库存数量
4. 可选择新的分区/状态（也可沿用原分区）
5. 系统更新对应商品的编码（数量段）和库存数量

### 5.2 出库流程

#### 5.2.1 批量出库

```
创建出库单 → 搜索选择商品 → 设置出库数量 → 确认生成出库单
                                              │
                                    ┌─────────┴─────────┐
                                    │                   │
                                取消出库             确认出库
                           (删除出库单,           (归档出库单,
                            恢复库存)             结束流程)
```

#### 5.2.2 预留出库

```
创建预留单 → 搜索选择商品 → 设置预留数量 → 确认生成预留单
                                              │
                                    ┌─────────┴─────────┐
                                    │                   │
                                取消预留             准备出库
                           (删除预留单,         (生成出库单,
                            恢复库存)        进入出库单流程)
```

**预留机制关键规则**：
- 创建预留单时，库存数量减少，预留数量增加
- 界面显示格式：`库存数量 + 预留数量`（如 `10 + 3`）
- 取消预留：删除预留单，库存数量恢复，预留数量减少
- 确认出库（准备出库）：生成出库单，预留数量减少，进入出库单流程

## 7. 数据库结构设计

### 7.1 技术方案

采用**微信云开发**提供的云数据库（文档型数据库），无需搭建独立服务器。

## 8. 图片存储方案

### 8.1 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. NAS 存储** | 无额外费用，空间大 | 需组网/内网穿透，上传下载速度受宽带限制 |
| **B. 微信云存储** | CDN 加速，稳定可靠 | 有免费额度限制，超出付费 |
| **C. 混合方案** | 兼顾成本与速度 | 架构复杂度增加 |

### 8.2 推荐方案：C. 混合方案

**核心思路**：上传时使用微信云存储作为中转，异步同步到 NAS；展示时优先从 NAS 读取，NAS 不可用时回退到云存储。

```
┌───────────┐     上传      ┌──────────────┐    异步同步    ┌─────────┐
│  微信小程序  │ ──────────→ │  微信云存储     │ ───────────→ │  NAS    │
│           │              │  (CDN 加速)   │              │  存储    │
│           │ ←─────────── │              │ ←─────────── │         │
└───────────┘     展示      └──────────────┘   定期清理      └─────────┘
                   优先 NAS，回退云存储
```

### 8.3 NAS 组网方案

由于 NAS 受限于宽带，推荐以下组网方案（按优先级排列）：

#### 方案一：Tailscale / ZeroTier 组网（推荐）

- 在 NAS 和使用端设备上安装 Tailscale 或 ZeroTier 客户端
- 形成虚拟局域网，无需公网 IP，无需端口映射
- NAS 端部署简单的文件接收服务（如 WebDAV 或自建 API）
- **优点**：免费、安全、配置简单、P2P 直连速度快
- **缺点**：需要每台访问设备安装客户端

#### 方案二：内网穿透（frp / ngrok）

- 在有公网 IP 的服务器上部署 frp 服务端
- NAS 端部署 frp 客户端，将文件服务端口映射到公网
- **优点**：无需客户端安装，小程序可直接访问
- **缺点**：需要一台公网服务器，速度受服务器带宽限制

#### 方案三：NAS 厂商自带远程访问

- 群晖：QuickConnect
- 威联通：myQNAPcloud
- **优点**：零配置
- **缺点**：速度受限，可靠性依赖厂商服务

## 11. 附录

### 11.1 编码生成示例

```javascript
/**
 * 生成商品编码
 * @param {string} inventoryId - 库存目录ID
 * @param {string} mainZone - 主分区 (A-Z)
 * @param {string} subZone - 子分区 (A-Z)
 * @param {number} quantity - 库存数量
 * @param {string} statusCode - 状态编码 (A-Z)
 * @returns {Promise<string>} 完整编码
 */
async function generateCode(inventoryId, mainZone, subZone, quantity, statusCode) {
  const db = wx.cloud.database();
  
  // 1. 检查回收池
  const recycled = await db.collection('recycled_seq_numbers')
    .where({ inventory_id: inventoryId, main_zone: mainZone, sub_zone: subZone })
    .orderBy('seq_number', 'asc')
    .limit(1)
    .get();
  
  let seqNumber;
  if (recycled.data.length > 0) {
    // 使用回收池中最小的序号
    seqNumber = recycled.data[0].seq_number;
    await db.collection('recycled_seq_numbers').doc(recycled.data[0]._id).remove();
  } else {
    // 自增序号
    const counter = await db.collection('seq_counters')
      .where({ inventory_id: inventoryId, main_zone: mainZone, sub_zone: subZone })
      .get();
    
    if (counter.data.length > 0) {
      seqNumber = counter.data[0].current_max + 1;
      await db.collection('seq_counters').doc(counter.data[0]._id).update({
        data: { current_max: seqNumber }
      });
    } else {
      seqNumber = 1;
      await db.collection('seq_counters').add({
        data: {
          inventory_id: inventoryId,
          main_zone: mainZone,
          sub_zone: subZone,
          current_max: 1
        }
      });
    }
  }
  
  // 2. 拼接编码
  const seqStr = String(seqNumber).padStart(4, '0');
  const qtyStr = String(quantity).padStart(4, '0');
  return `${mainZone}-${subZone}-${seqStr}-${qtyStr}-${statusCode}`;
}
```
